# 九区正式 Agent 实验：P0 无政策基线

## 目的

本入口把两区简化机制迁移到正式九区连续城市。Z1–Z9是功能分区和OD统计单元，不是九个独立城镇。第一轮用于验证人口、活动、空间、四种交通方式、道路反馈和网约车车辆守恒能否在同一入口中闭合。

## 连续城市与功能分区

- 所有区域都有内部道路、步行和公交网络。
- 公交线路和地铁线路跨越功能区边界，区域边界不阻断交通。
- 功能分区决定人口起点、活动吸引、站点覆盖和出行流向。
- W0/W1/W2使用完全相同的Agent、活动和OD。
- 本基线关闭天气取消和remote work，以便先单独观察交通效应。

## 四种交通方式

```text
walk
bus
metro
ride_hailing
```

公交继承九区B1–B4、区内服务、跨区换乘和早晚高峰班次。网约车继承道路时间、费用与九区日初整数车辆池。步行受最大距离限制，不计入道路车辆。

地铁采用三条连续跨区主干线：

```text
M1：Z8—Z2—Z1—Z7—Z4
M2：Z6—Z3—Z1
M3：Z3—Z5—Z7
```

Z1、Z2、Z3、Z7站点较密，满足距离与稳定覆盖抽样时允许区内地铁。Z4、Z5、Z6、Z8有跨区乘车站点，但普通区内OD不直接提供地铁。Z9无地铁，通过公交接驳Z6。

## Agent如何选择

```text
utility
= - generalized_cost_weight × (时间价值 × 总时间 + 费用)
   + 年龄方式常数
   + 当前天气窗口的方式偏好
   + 固定seed的Gumbel扰动
```

天气偏好只能改变效用，不能指定方式。非数字接入且无`family_assistance`者不能独立选择网约车。地铁根据实际起终点、站点覆盖、运营时间、接驳、候车和换乘参与效用比较。

## 交通反馈

公交班次车辆和第一轮成功派单网约车形成代表性道路流量，同时影响公交与网约车车内时间；随后Agent重新选择一次。地铁不增加道路车辆流量，其速度与班次不受天气和道路拥堵影响，但早晚高峰会增加班次。

网约车按实际请求时间、固定派单优先值和leg_id排序。车辆行程期间保持busy，到达后留在目的地区域。起点无车或等待超限后最多fallback一次。

## 输出与边界

输出Agent、活动状态、计划legs、四方式选择、派单审计、期末车辆状态和六行宏观汇总。当前道路反馈仍是全市代表性共享走廊，不是路段级分配；所有参数用于机制实验，不是上海实测值或现实预测。
