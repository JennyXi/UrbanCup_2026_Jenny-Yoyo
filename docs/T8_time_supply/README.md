# T8：正常天气分时段基础交通供给层

本层叠加在T7静态九区交通网络之上，只描述正常天气下、外生给定的分时交通供给。它不修改静态OD表，也不读取Agent年龄、数字接入或偏好，不实现天气、选择反馈、内生拥堵、动态加价、车辆短缺或派单。

## 八个时间段

时间段统一配置在`config/time_dependent_transport_supply.json`，函数中不保存时段边界。

| period_id | 中文 | 时间 | 强度 |
|---|---|---|---|
| `morning_shoulder` | 早间肩部 | 07:00–08:00 | 轻度高峰 |
| `morning_core_peak` | 早核心高峰 | 08:00–09:30 | 核心高峰 |
| `morning_recovery` | 早间回落 | 09:30–10:30 | 轻度高峰 |
| `day_off_peak` | 日间平峰 | 10:30–17:00 | 平峰 |
| `evening_shoulder` | 晚间肩部 | 17:00–17:30 | 轻度高峰 |
| `evening_core_peak` | 晚核心高峰 | 17:30–19:00 | 核心高峰 |
| `evening_recovery` | 晚间回落 | 19:00–20:00 | 轻度高峰 |
| `night` | 夜间 | 20:00–次日07:00 | 夜间 |

边界采用左闭右开。例如08:00已经属于早核心高峰，09:30属于早间回落。配置校验要求八段恰好覆盖一天的每一分钟，不能重叠或留空。

## 空间层级

- 强供给区：Z1、Z2、Z3、Z7；
- 普通外围：Z4、Z5、Z6、Z8；
- 远郊弱供给：Z9；
- 就业节点组：Z1、Z7、Z6；
- 通勤外围组：Z4、Z5、Z8、Z9。

本层不改变T7拓扑。道路、公交和网约车仍覆盖九区；地铁仍使用M1–M3；Z9仍先乘公交到Z6再进入地铁。区内距离、区内地铁覆盖率和静态接驳结构继续来自T7。

## 各方式参数

### walk

全天可用，速度倍率为1.00，保留T7的6 km最大距离。本轮不加入天气、年龄、夜间安全或个人偏好。

## 方向性高峰

方向规则按区域组配置，不枚举单个OD：

- 早高峰：通勤外围组→就业节点组；
- 晚高峰：就业节点组→通勤外围组。

八个城市基础时段本身不移动，只对上述主方向的负荷窗口增加小幅相位偏移：

| 方向负荷 | 普通外围Z4/Z5/Z8 | 远郊Z9 |
|---|---:|---:|
| 早间外围出发 | 提前15分钟 | 提前30分钟 |
| 晚间外围到达 | 延后15分钟 | 延后30分钟 |

因此普通外围早核心方向负荷约为07:45–09:15，Z9约为07:30–09:00；普通外围晚核心方向负荷约为17:45–19:15，Z9约为18:00–19:30。肩部和回落会随同方向窗口一起小幅平移，但`time_period`仍按原八段输出。例如07:45仍是`morning_shoulder`，只是外围→就业节点的方向负荷已经进入核心强度。该方向的公交和地铁同时采用核心高峰班距，即人流增加时班次加密，而不是只增加拥挤：公交核心基准班距6分钟、地铁3.5分钟，再乘区域供给倍率。偏移被配置校验限制在30分钟以内，避免与已有肩部—核心—回落趋势重复放大。

公交和网约车共用同一套道路速度规则：非高峰最终乘数1.00，普通高峰最终乘数0.85，最强通勤方向核心高峰最终乘数0.75。0.75会直接替换0.85，绝不进行`0.85 × 0.75`叠乘。公交基础18 km/h对应15.3和13.5 km/h；网约车基础33 km/h对应28.05和24.75 km/h。步行与地铁最终速度乘数恒为1.00。主方向等待负担仍按方式分别设置；地铁主方向可增加等待和换乘负担，但不降低车内速度。

## 跨时段计算

接驳结束后才开始候车。候车若跨越实际时段边界或郊区方向性偏移边界，会按两侧时段的期望候车服务率分段完成，不再把整段等待锁定在开始时刻。随后把基础车内时间理解为“待完成的基础运行工作量”：

```text
一个实际分钟完成的基础车内分钟 = 当前时段速度倍率
```

如果行程跨过时段边界，就在边界处切段，以新时段倍率继续消耗剩余工作量。因此07:50–08:10会精确记录早肩部10分钟和早核心10分钟，而不是只按出发时段计算整段。`period_speed_multiplier`是整段车内运行的有效倍率。`time_period`按实际调整后完整leg所覆盖的时段顺序输出，例如`morning_shoulder+morning_core_peak`。

地铁线路换乘惩罚按车内分段分钟加权。Z9公交接驳根据它实际发生的时间单独分段；公交—地铁方式转换仍只加一次6分钟。

## 地铁末班车

系统对每个具体OD从服务截止时刻向前反推：

```text
latest_feasible_departure
+ 分时接驳、候车、车内和换乘时间
<= 23:30服务截止
```

因此长距离、换乘和Z9接驳OD的最晚可行出发时间早于短距离直达OD。若具体leg出发时间超过该值，静态地铁`available`仍保留为true，但分时层`operating=false`、`supply_level=unavailable`，避免覆盖原始OD供给事实。

## 输出

`scripts/generate_50_agent_spatial_example.py`保留原`leg_mode_options.csv`，另生成`leg_mode_time_supply.csv`。新文件保留所有静态字段，并增加：

```text
time_period
operating
base_total_time_min
period_speed_multiplier
period_wait_time_min
period_transfer_penalty_min
time_adjusted_total_time_min
latest_feasible_departure
supply_level
```

`departure_time`同时写入新文件，便于独立审计时段判断。本层不改变fare，也不选择方式；每条leg都根据自己的出发时间和方向重新计算速度，去程结果绝不复用于返程。同一个Agent的去程和返程既可能处在不同高峰状态，未来也可以选择不同交通方式。

各层语义严格分离：T7的`base_speed_kmh`只表示正常交通条件；T8只选择1.00、0.85或0.75中的一个时段道路乘数；天气乘数由后续天气层额外处理；选择反馈产生的动态拥堵、车辆占用、运力短缺和派单等待由后续供需模块处理。本层不包含后三类因素。
