# 两区简化 Agent 交通方式选择实验

这是 `Simple_Tests` 分支的最小实验。空间和交通网络被压缩，但天气、Agent 异质性、活动出行语义和可审计输出继续遵循 `main` 的设计边界。

## 空间和交通

- `S1`：中心就业区，公交覆盖率 `0.90`，接驳和等待时间较短。
- `S2`：外围混合居住区，公交覆盖率 `0.55`，接驳和等待时间更长。
- 唯一公交线路 `B1` 连接 `S1—S2`。
- 区内和跨区均可选择 `walk`、`bus`、`ride_hailing`。
- 网约车采用普通快车模拟基准：最低消费 14 元、2.3 元/km、0.5 元/min，20 km 以上部分再加 0.8 元/km；最终费用取最低消费与计价费用中的较大值。
- 天气动态价格乘数为 W0=1.00、W1=1.05、W2=1.15。它是供需情景假设，不表示恶劣天气必然按固定比例加价。
- 上海网约车实行市场调节价；这些参数只用于模型基准和敏感性分析，不解释为滴滴 2026 年官方统一价格。
- 同区出行使用大于 0 的区内平均距离；S1—S2 合成道路距离约 4.86 km，因此三种方式均可行。

公交覆盖率不直接随机删除公交选项。它通过不同的接驳时间和等待时间表示服务差异：S2 覆盖较低，因此同样距离下的公交广义成本更高。这与 `main` 中 `zone_service_parameters` 的分区供给思想一致。

## 跟随 main 的实验设计

- Agent 年龄层仍为 `18–39`、`40–59`、`60+`。
- 数字接入不足的 Agent 不能独立选择网约车。
- 天气仍为 `W0` 正常、`W1` 极端高温、`W2` 暴雨。
- 暴雨降低三种方式速度并增加公交、网约车等待；高温主要降低步行吸引力。天气只改变时间和效用，不直接禁用步行。
- 方式效用由旅行时间、费用、天气、年龄偏好和固定 seed 的稳定随机项组成。
- 数字接入或家庭协助表示网约车可用，不保证 Agent 会选择网约车。
- 方式选择前复用 `main` 的 T2：按天气暴露、年龄、活动目的、行动约束和时间灵活性判断活动是否取消；取消活动同时移除外出和返程两条 leg。
- 网约车采用一次需求反馈：先完成第一轮选择，以初始请求数增加等待时间，再使用相同随机项重新选择一次；不继续迭代。
- 每次选择保留全部可行备选项、服务覆盖率和效用，便于后续接入 T2 活动取消、T3 补贴、T8/T9 供给及 T10 拥堵。

建议第一轮使用 50 个 Agent，分别运行 W0、W1、W2；对比年龄组、居住区和数字接入分组的方式份额、平均旅行时间与平均费用。当前参数均为 MVP 假设，不是实证估计。

## 50-Agent 实验

50 人的年龄与身份比例直接复用 `main` 的 T1 人口生成器。默认按年龄层稳定分配为 S1 40%、S2 60%，每人生成一条外出和一条返程 leg，并依次运行 W0、W1、W2。

```powershell
python -B -X utf8 -m scripts.run_50_agent_experiment
```

结果写入 `outputs/simple_agent_50/`：

- `activity_weather_decisions.csv`：每个 Agent 在 W0/W1/W2 下的活动继续或取消决定；
- `agent_mode_choices.csv`：仅保留实际执行 leg 的方式选择明细，因此接入取消机制后少于或等于 300 条；
- `system_impact_summary.csv`：三种天气的方式份额、公交负荷、网约车 VKT 与道路 PCU 汇总；
- `experiment_summary.json`：人口比例和系统汇总。

这里的 PCU 是代表日网约车车辆数，不能直接作为 T10 的 `PCU/hour/direction`。后续接入 T10 前必须按方向和时间窗聚合。

网约车需求反馈参数位于 `config/simple_agent_model.json` 的 `ride_hailing_demand_feedback`。当前每个第一轮请求增加 0.12 分钟等待，额外等待上限为 12 分钟，并固定只迭代一次。
